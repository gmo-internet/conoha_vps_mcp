# テストグループ定義

全37テストケースを19グループ（A〜S）に分類し、依存関係に基づいて順次実行する。

## グループ一覧

### Group A: サーバー基本操作

- **テストNo.**: 1 → 2 → 3
- **内容**: サーバー作成 → サーバー停止 → サーバー起動
- **効率化**: No.3の実行完了時点でNo.1, 2, 3すべてを確認できる
- **クリーンアップ**: サーバー削除 → ボリューム削除

### Group B: サーバー強制停止

- **テストNo.**: 1 → 4
- **内容**: サーバー作成 → サーバー強制停止
- **クリーンアップ**: サーバー削除 → ボリューム削除

### Group C: サーバー再起動

- **テストNo.**: 1 → 5
- **内容**: サーバー作成 → サーバー再起動
- **クリーンアップ**: サーバー削除 → ボリューム削除

### Group D: サーバーリサイズ確定

- **テストNo.**: 1 → 2 → 6 → 7
- **内容**: サーバー作成 → サーバー停止 → リサイズ（4GB） → リサイズ承認
- **効率化**: No.7の実行完了時点でNo.6, 7を確認できる
- **注意**: リサイズ後、ステータスが `VERIFY_RESIZE` になるまで待機が必要
- **クリーンアップ**: サーバー停止 → サーバー削除 → ボリューム削除

### Group E: サーバーリサイズ取消

- **テストNo.**: 1 → 2 → 6 → 8
- **内容**: サーバー作成 → サーバー停止 → リサイズ（4GB） → リサイズ取消
- **注意**: リサイズ後、ステータスが `VERIFY_RESIZE` になるまで待機が必要
- **クリーンアップ**: サーバー削除 → ボリューム削除

### Group F: ボリューム更新

- **テストNo.**: 1 → 9
- **内容**: サーバー作成 → ボリューム名を「aaa」に変更
- **クリーンアップ**: サーバー削除 → ボリューム削除

### Group G: セキュリティグループ＋サーバー削除

- **テストNo.**: 1 → 10 → 13 → 17
- **内容**: サーバー作成 → SG作成（web-server-secgroup） → SGをサーバーポートに設定 → サーバー・ボリューム・SG削除
- **効率化**: No.17の実行でNo.10, 13, 17すべてを確認できる
- **クリーンアップ**: No.17自体がクリーンアップを兼ねる（自己完結）

### Group H: セキュリティグループ更新

- **テストNo.**: 10 → 11
- **内容**: SG作成（web-server-secgroup） → HTTPルール削除＋SG名変更（https-secgroup）
- **クリーンアップ**: セキュリティグループ削除（https-secgroup）

### Group I: セキュリティグループ詳細確認

- **テストNo.**: 10 → 12
- **内容**: SG作成（web-server-secgroup） → SG詳細＋HTTPSルール詳細表示
- **クリーンアップ**: セキュリティグループ削除（web-server-secgroup）

### Group J: サーバー紐付き情報

### Group R: 異常系テスト

- **テストNo.**: 24, 25, 26, 27, 28, 29, 30
- **内容**:
  - No.24: パスワード条件違反（`12345`）
  - No.25: サーバー名条件違反（`+test`）
  - No.26: SSHキー名条件違反（`+ssh_key`）
  - No.27: ボリューム名条件違反：作成（`+test`）
  - No.28: ボリューム名条件違反：更新（`+aaa`）→ サーバー作成が前提
  - No.29: アタッチ中ボリューム削除 → サーバー作成が前提
  - No.30: Prompt実行パスワード条件違反（`aaa`）→ `conoha_post` path=`/servers` で `adminPass=aaa` をバリデーション検証（No.24と同様の手法）。**スキップ禁止**
- **実行順序**:
  1. No.24, 25, 26, 27, 30 を順次実行（リソース不要、すべてバリデーションエラー確認）
  2. サーバー作成（No.1相当）
  3. No.28, 29 を実行
  4. サーバー・ボリューム削除
- **クリーンアップ**: No.28, 29 実行後にサーバー・ボリューム削除

- **テストNo.**: 31, 32, 33, 34, 35, 36, 37
- **内容**:

- **クリーンアップ**: オブジェクト削除 → コンテナ削除

## 共通注意事項

### サーバー作成の共通手順

多くのグループでNo.1（サーバー作成）を前提とする。手順:

1. `conoha_get` path=`/flavors/detail` → メモリ1GBのフレーバーID取得
2. `conoha_get` path=`/v2/images?limit=200` → Ubuntu 24.04のimageRef取得
3. `conoha_get` path=`/types` → ボリュームタイプ取得
4. `conoha_post` path=`/volumes` → ボリューム作成（size:30）
5. `conoha_post` path=`/servers` → サーバー作成
6. ステータスがACTIVEになるまで待機

### サーバー・ボリューム削除の共通手順

1. `conoha_get` path=`/servers/detail` → サーバーID確認
2. `conoha_get` path=`/volumes/detail` → ボリュームID確認
3. `conoha_delete_by_param` path=`/servers` param=サーバーID
4. サーバー削除完了を待機
5. `conoha_delete_by_param` path=`/volumes` param=ボリュームID
